# Lifecycle Stages

## discovery

The work is exploring a problem, audience, opportunity, or product direction.

Expected evidence: problem statement, target users, market signals, user pain, constraints, decision needed.

## definition

The work is defining scope, requirements, PRD, positioning, business model, or launch goals.

Expected evidence: requirements, acceptance criteria, scope boundaries, user scenarios, success metrics, business assumptions.

## design

The work is shaping UX, interaction flows, service flows, agent workflows, or product behavior.

Expected evidence: user flows, screens, interaction states, workflow diagrams, edge states, errors, accessibility expectations.

## architecture

The work is defining technical boundaries, data flow, APIs, storage, dependencies, infrastructure, security model, or agent orchestration.

Expected evidence: architecture notes, repo structure, interfaces, configs, dependency manifests, data flows, permission model.

## implementation

The work is code, diff, refactor, bugfix, integration, migration, prompt/workflow implementation, or test changes.

Expected evidence: git diff, changed files, nearby tests, related modules, error output, constraints.

## verification

The work is validation, QA, eval, release confidence, regression testing, or acceptance review.

Expected evidence: test plans, eval datasets, test results, CI output, acceptance criteria, known risks.

## launch

The work is preparing for release, GTM, rollout, customer communication, operational readiness, or pricing/package changes.

Expected evidence: release plan, rollout plan, messaging, pricing, support plan, monitoring, rollback plan.

## operation

The work is running in production or market, handling support, metrics, incidents, growth loops, reliability, or customer feedback.

Expected evidence: logs, metrics, support feedback, alerts, runbooks, funnel data, customer signals.

## postmortem

The work is incident review, product retro, launch retro, failed experiment, or operational learning.

Expected evidence: timeline, impact, root cause evidence, contributing factors, remediation, owner follow-up.

## unknown

Use `unknown` only when evidence is insufficient to classify the stage. If stage is unknown, select a small role set and make the uncertainty explicit.
